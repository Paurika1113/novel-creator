# Novel Creator Codex 重设计方案

> 2026-06-10 | 分支 `skill-refactor`
> 参考项目：DeepSeek-GUI Write Mode、Accomplish、AI-Novel-Writing-Assistant

---

## 一、设计目标

将当前"多页面 IDE 工具"改造为**类 Codex 的统一创作工作台**——作者在单一窗口中完成从灵感到大纲到正文到归档的全部工作流，AI 始终在侧，按需响应。

核心原则：
- **内容居中** — 正在写的文字永远是视觉焦点
- **AI 在侧** — 助手面板常驻但不抢占注意力
- **指令自然** — 斜杠命令 + 自然语言，无需切换"Agent 人格"
- **进度可见** — 章节编号、字数、上下文预算始终可感知

## 四、斜杠命令系统

替换 ActionBar 按钮，在输入框输入 `/` 自动呼出命令菜单。

### 4.1 命令列表

| 命令 | 标签 | 说明 | 需要条件 |
|------|------|------|---------|
| `/outline` | 生成大纲 | 为下一章生成详细章节大纲 | 有总纲 |
| `/write` | 写新章 | 根据大纲撰写全新章节正文 | 有章纲 |
| `/continue` | 续写 | 在草稿末尾追加内容 | 有草稿 |
| `/review` | 审核草稿 | 五维审核当前草稿质量 | 有草稿 |
| `/polish` | 润色文风 | 优化当前草稿的语言风格 | 有草稿 |
| `/world` | 世界观 | 从已归档章节提取/更新世界观 | 有归档章节 |
| `/summarize` | 摘要 | 为已归档章节生成摘要 | 有归档章节 |
| `/archive` | 归档章节 | 归档当前草稿为正式章节 | 有草稿 |

### 4.2 交互行为

```
用户在输入框输入 / → 弹出命令菜单
  ├── 键盘 ↑↓ 选择，Enter 确认
  ├── 鼠标点击选择
  └── 选择后：输入框预填命令 prompt，用户可修改后发送
```

### 4.3 实现方式

- 复用 `SKILLS` 常量，将 `SkillDef.prompt` 作为默认指令
- 用户可额外添加文字（如 `/write 请使用更悬疑的风格`）
- 不需要后端改动，纯粹的 UI 交互增强

## 五、编辑器视图模式

### 5.1 Source 模式（纯源码）

当前 `MarkdownEditor.tsx` 已实现。textarea 直接编辑 Markdown 源码。

### 5.2 Split 模式（源码 + 预览）

左半边：可编辑的 textarea
右半边：实时渲染的 Markdown HTML
两边同步滚动（相互关联）

### 5.3 Preview 模式（只读预览）

只渲染 Markdown HTML，不可编辑。用于审阅已归档章节。

### 5.4 实现方案

- 渲染引擎：复用 `renderMarkdown()` 现有逻辑
- 滚动同步：监听 scroll 事件，计算百分比映射
- 模式状态：存入 `useEditorStore.viewMode: 'source' | 'split' | 'preview'`

## 七、风险与约束

| 风险 | 缓解 |
|------|------|
| 大规模 UI 重写可能引入回归 bug | Phase 分步提交，每步构建验证 |
| 编辑器分屏渲染性能 | 使用 `useMemo` + `requestAnimationFrame` 防抖 |
| 导航栏复杂度增加（合并多页面功能） | 保持各 section 组件独立，用条件渲染 |
| 用户可能不适应新布局 | 保留原有布局作为可选"经典模式" |

## 八、技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 导航栏数据源 | `useEditorStore.filesByBook` + `useBookStore.books` | 已有数据，无需新增 store |
| Markdown 渲染 | 保持正则渲染 | 已稳定，不引入 marked/remark 依赖 |
| 命令菜单实现 | 纯 CSS + React state | 不引入第三方 command palette 库 |
| 状态栏 | 新建独立组件 | 职责单一，方便后续扩展 |
| 视图模式存储 | `useEditorStore.viewMode` | 与编辑器状态内聚 |
